# Worked Example - Closed Loop

## Status
Diese Seite bleibt vorerst ein Template, jetzt aber als expliziteres Proof-Schema.

## Proof-Ziel
Ein kuenftiges Worked Example sollte eine enge Aussage auditierbar machen:

> ein reviewed signal fuehrte zu einer reversiblen runtime-wirksamen Aenderung und spaeter zu einer gemessenen Wirkung.

## Kanonische Objekt-Spine
Das Example sollte die verknuepfte ID-Kette aus `trace-spine.md` und `data/trace-spine-schema.json` verwenden:
- `signal_id`
- `review_id`
- `adoption_id`
- `change_id`
- `measurement_id`

## Erforderliche Abschnitte
### 1. Signal
- Was ist passiert?
- Warum galt es als stark genug?
- Welche Evidenzklasse gilt?

### 2. Review
- Wer hat reviewed?
- Welche Alternativen wurden betrachtet?
- Warum ging es weiter?

### 3. Adopted change
- Was hat sich exakt geaendert?
- War die Aenderung reversibel?
- Welche Wirkung wurde erwartet?

### 4. Runtime trace
- Wo ist die Aenderung sichtbar?
- Wann wurde sie aktiv?
- Welche Systemoberflaeche zeichnet sie auf?

### 5. Measurement
- Welches Beobachtungsfenster wurde verwendet?
- Welche Metrik oder welches qualitative Signal wurde geprueft?
- Was aenderte sich danach?

### 6. Unsicherheit und Konfundierungen
- Was koennte das Ergebnis sonst erklaeren?
- Was bleibt unbewiesen?
- Was waere eine Widerlegung?

## Mindest-Verdict-Felder
- `claim_status`
- `confidence_posture`
- `review_date`
- `limitations`
- `follow_up_needed`
